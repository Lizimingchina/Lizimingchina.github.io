WSL(WINDOWS下的linux子系统)

发表于 2019-03-30


WINDOWS10的Linux子系统



摘要:在Windows10系统中，增加了一项黑科技：Linux子系统，即Windows Subsystem for Linux，以下简称”WSL“。本文主要介绍了WSL的安装使用方法，以及一些比较基础的操作。

关键词 :WSL、新手入门。

1.前言

作为刚接触Linux的新手，我们首要体验Linux的方式是通过VMVare等软件安装虚拟机，再使用图形界面操作、练习命令；或者有些爱折腾的同学会选择双系统。前者通常比较占用内存资源，后者两个系统来回切换，也比较麻烦。而Windows Subsystem for Linux (以下简称WSL)为我们提供了一种新的选择。它能够在比较小的资源占用下，让我们体验到原生的Linux功能。本文将从安装入手，用图文一步步说明，起到一定的参考作用。

2.具体流程

原理了解

可能大家比较好奇，Windows系统是闭源且有版权的系统，Linux是开源系统；而且双方的构成框架肯定不一样，是怎样同时存在的呢？

实际上，WSL通过在Windows NT内核上模拟Linux内核接口，简单来说，安装的Linux系统实际上是以Windows NT内核为基础，进行接口封装，来模拟Linux内核。

拿文件系统举例。

Linux上的文件系统

Linux通过虚拟文件系统（VFS）抽象文件系统操作，虚拟文件系统为用户模式程序提供了与文件系统交互的接口（通过系统调用，如open，read，chmod，stat等）。VFS为用户提供所有文件系统的单一命名空间视图，这能够允许多个文件系统共存，提供相同的操作和语义。

在Linux中，文件系统安装在命名空间中的不同目录中。例如，硬盘驱动器可以安装在根目录/上，目录如/dev，/proc，/sys和/mnt 、/cdrom都可以安装不同的文件系统——这些文件系统可以位于不同的设备上。

Windows上的文件系统

Windows将所有系统资源概括为对象——不仅包括文件，还包括线程、共享内存部分和计时器等。所有打开文件的请求最终都通过NT内核中的对象管理器进行处理，该内核将请求通过I / O管理器路由到正确的文件系统驱动程序。文件系统驱动程序在Windows中实现的接口更简单，而且执行的要求更少。比方说，没有像Linux那样的inode结构或类似的东西;

Windows中的文件系统通常安装在驱动器号上，如C：，D：等，尽管它们也可以安装在其他文件系统的目录中。这些驱动器号实际上是Win32的构造，而不是对象管理器直接处理的东西。对象管理器保留一个类似于Linux文件系统命名空间的命名方式，举个例子，F:\face2face\ours就是我们经常在资源管理器里看到的路径（目录层次）。

WSL中的文件系统

文件系统的模拟就是关键，Windows的Linux子系统必须将各种Linux文件系统操作转换为NT内核操作。WSL必须提供一个Linux系统文件可以存在的地方，其中包含所有必需的功能，比如Linux权限、符号链接和其他特殊文件，如FIFO。下面有一张官方的解释图：

wsl文件系统

从上图我们可以知道当应用程序调用系统调用时，这由系统调用层处理，系统调用层定义各种内核入口点，如open、read、chmod、stat等。对于这些与文件相关的系统调用，系统调用层其实没有过地处理， 它基本上只是将调用请求向下转发给VFS。最终还是由NT内核实现。

系统安装

既然是模拟，所以相比虚拟机和双系统而言，他比虚拟机占用更少的资源，但无法像双系统那样完整利用硬件资源。由于面世不久，wsl存在一些问题，目前来看还是“玩具”级别的，仅供新人玩玩还是足够的。在简单了解wsl的模拟机制后，接下来进行wsl的安装与应用。

4.结论

本文从最开始的安装，到最后的桌面环境尝试，给出了较为详细的讲解步骤。在文章的开头，就使用文件系统的模拟来介绍，说到底WSL还是利用NT内核进行的调用封装，但这并不影响新手的正常使用。

从便利性的角度，采用WSL方式其实比安装虚拟机或者双系统更加方便，整个过程比较傻瓜式。如果对图像界面有要求的话（我就是不习惯字符界面），按照本文几分钟就可以配置一个桌面环境；从资源占用的角度上来讲，WSL占用了较低的硬件资源，反应也较为迅速。
